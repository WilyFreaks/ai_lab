# AI Lab - エージェント規約（Splunk ワークショップ）

**ワークショップの目的:** 本ラボは、**インテリジェントエージェントにとっての Splunk の価値（機械データプラットフォーム）**を示すことを意図している。**多様な運用データ**（合成 UX メトリクス、TWAMP、ストリーミング telemetry、IOS／syslog、WDM PM／アラート、スケジュール検索の成果物など）を **同一タイムライン上で相関**し、不透明な要約ではなく **明示的な SPL とナレッジオブジェクト**（index／sourcetype、保存済みサーチ、ルックアップ、時間窓）で説明できるようにする。エージェントの応答は **Splunk がデータを取り込み・モデル化・検索する過程**が伝わるようにし、ファシリテータが **「エージェント＋Splunk」** のストーリーを語れるようにする（根拠のない推測話にしない）。

**ai_lab** の 5G WDM バックボーン向けワークショップで、エージェントが Splunk をどう扱うかの規約です。リポジトリの `.cursor` 規約および `docs/project_ai_lab.md`、`docs/project_test_design.md` と併用します。

---

## 1. Splunk へのアクセス

1. **`splunk-mcp-server` が利用可能なら Splunk MCP を優先**する。ツール呼び出し前に MCP ツール定義（スキーマ）を確認する。
2. **認証:** REST または CLI の **`-token`** には **Bearer トークン**を使う（例: Cursor MCP の `AUTH_TOKEN`）。パスワードの推測はしない。
3. MCP が使えない場合は **`splunk` CLI** または管理ポートへの REST にフォールバックし、可能ならトークン認証を維持する。

---

## 2. クエリと保存済みサーチ

1. **保存済みサーチ優先:** ワークショップの検証・調査では **`ai_lab`** アプリの **`| savedsearch <名前>`** を優先する。スタンザ名は `default/savedsearches.conf` を参照し、環境で上書きされている場合は **`local/savedsearches.conf`** も確認する。現在出荷済みの **32 件すべて**に `description =` フィールドがある — 呼び出す前に確認する。
2. ユーザーが明示しない限り、**他アプリの保存済みサーチは実行しない**。
3. 適切な保存済みサーチがない場合のみ **生の SPL** を使い、**`index=` / `sourcetype=`** をデータ契約に合わせて明示する。
4. **エピソード / アラート相関:** マテリアライズ済みのワークショップ用オブジェクトがあれば優先（例: **`index=episode`**）。デプロイに **`list_episodes`** 等が **`ai_lab` 内で定義されていれば** それを使う。なければシナリオ用の **`index=alerts sourcetype=ai_lab_alert`** でアラート行を参照する（現行契約はプロジェクトドキュメント参照）。
5. **予測（フォーキャスト）保存済みサーチ:** `forecast_cdtsm` / `forecast_predict`（ThousandEyes 単一系列）、`forecast_cdtsm_multi_series` / `forecast_predict_multi_series`（全ルータのインタフェース別トラフィック）が出荷済み。異常検知や ML デモに活用する。
6. **パラメータ付き保存済みサーチ:** `cnc_interface_ifInPktsRate_for_a_router` は `router` 引数が必要（例: `router="R2"`）。MCP の `args` または CLI トークン置換で渡す。`description =` にパラメータ仕様が記載されている。
7. **`description =` を使った保存済みサーチの選択方法:** `ai_lab` の各保存済みサーチの `description =` は次の構造で記述する：
   - **ケイパビリティ（先頭）** — 返されるフィールド、データソース、index/sourcetype。これが検索選択の主なシグナル。この記述が自分の質問に答えられるか確認する。
   - **「ユースケース例:」（末尾）** — 具体的な調査トリガー（例:「迂回の確認」「スライス別パケットロスの確認」）。ヒントであり、網羅的なリストではない。記載されていないケースでも利用してよい。
   - **「ユースケース例」に列挙されていないから使えない、とは判断しない。** ケイパビリティが自分のニーズに合致していれば呼び出してよい。
   - `description =` がない保存済みサーチは `search =` の SPL を直接読んでケイパビリティを推定してから、呼び出すか生 SPL を書くかを判断する。

## 3. データ契約（違反禁止）

1. **TWAMP（`pca_twamp_csv`）:** **`ul_lostperc` / `dl_lostperc`** は **0-100 の整数パーセント**（百万分率などではない）。レート系は仕様上 **pps** として扱う。UL シーケンス等はスクリプト設計ドキュメントに従う。
2. **Telemetry（`cnc_interface_counter_json`）:** ワークショップのルックアップと保存済みサーチのフィールド名に従う（例: **`telemetry_if_counter`**、IF レート系保存済みサーチ）。
3. **WDM PM（`wdm_pm`）:** **`LSBIASCUR` / `SUMOOPCUR`** は **Tx 側寄り**、**`FEC_BEF_COR_ER` / `SUMIOPCUR`** は **Rx 側寄り**として扱う。エンドポイントは **`lookups/router_wdm_transponders.csv`** で束ねる（`docs/project_ai_lab.md`）。
4. **ThousandEyes:** アプリの sourcetype / フィールドに従い、境界値と比較する際は **単位（ms と s 等）** に注意する。

---

## 4. ワークショップのトーンと根拠

1. 既定では **根拠と仮説**（時刻相関、影響スライス／リンク）までにとどめる。ユーザーが解説を求めるか、データ上の鎖が明らかでない限り、**単一の決定的「根本原因」表現**は避ける。
2. **シナリオ／クイズモード:** ユーザーが求めない限り、参加者向けに **答えの丸投げ** をしない。
3. **「復旧」/ SR-TE:** Splunk 上で観測できる **経路・ポリシー信号**（IOS、telemetry、SR-TE JSON）に留める。実ネットワークへの設定投入をエージェントが行ったかのような表現は、スコープに含まれる場合のみに限定する。

---

## 5. ワークショップ環境

1. **データは事前に準備済み:** ワークショップの Splunk インスタンスには、すべての index・sourcetype・保存済みサーチ・ルックアップが揃っている。データの投入や変更は不要。保存済みサーチでクエリするだけでよい。
2. **シナリオ状態:** 障害シナリオはファシリテータが `scenario_control` ダッシュボードで制御する。シナリオ設定を直接変更しない。有効化やリセットが必要な場合はファシリテータに確認する。
3. **スコープ:** クエリと調査は **`ai_lab`** アプリの index（`telemetry`、`twamp`、`thousandeyes`、`syslog`、`ios`、`alerts`）に限定する。ファシリテータの明示的な指示がない限り、他アプリの index は参照しない。

---

## 6. 調査スキル（`skills_en.md` / `skills_jp.md`）を使うタイミング

**5G WDM バックボーン障害調査**のワークフロー（ThousandEyes + アラート + TWAMP + telemetry + IOS + WDM の横断）は、ユーザーの意図が **マルチソース相関** に合致するときに使う。例:

- WDM / バックボーンの **障害調査**、**パケットロス**、**サービス劣化**、**迂回 / SR-TE 経路**、**エピソード／アラート相関**、**「TWAMP 品質の調査」など文脈付きの TWAMP 調査**

**強制しない:** ジェネレータコード修正、**`scripts/test_*.sh`** のデバッグ、保存済みサーチのパッケージングのみ、汎用 Splunk 管理 - 通常の開発規約に従う。

---

## 7. トリガー（厳密な文字列マッチではなく指針）

- **強いトリガー:** 「バックボーン障害を調べて」「WDM とルータを相関」「エピソード／アラートの時系列」「スライス横断のパケットロス」「SR-TE 迂回の話」など。
- **弱いトリガー:** 単体の **「TWAMP」** だけ - 全文が **エンドツーエンド調査** かどうか確認し、狭い質問ならフル 7 ステップに入らず回答する。

---

## 8. UX（任意）

ユーザーが **コンパクト実行**（バナー省略）を求めた場合はステータス行を短くしてよいが、**ステップ境界**と**サマリー**は残し、追検証できるようにする。

## 9. Cursor Canvas - データ駆動のワークショップ用ボード

ファシリテータが **A/B やスコアボード**（例: **LLM コストと保存済みサーチ再利用**、**レイテンシ**、**精度**）を見せたいときは、長い説明だけのチャットより **Cursor Canvas**（チャット横の単一 `*.canvas.tsx`）を優先する。

1. **配置:** Canvas は次の配下にのみ作成する:  
   `~/.cursor/projects/<ワークスペース>/canvases/<名前>.canvas.tsx`  
   （Cursor の canvas スキル: `~/.cursor/skills-cursor/canvas/SKILL.md` 参照）。`cursor/canvas` **以外から import しない**。**`fetch` 禁止**、追加モジュール禁止 - データはスキルに従い **埋め込みまたは永続化**する。

2. **Splunk 由来の数値:** Splunk で測った値（ジョブ時間、行数、検証用の一致など）は **定数または表**にし、**いつ・どう測ったか**（`recordedAt`、REST oneshot / MCP 往復など）を明記する。デモ前に数値を頼む場合は **開催直前に定数を更新**する。

3. **LLM トークンとアシスタントのターン遅延:** エージェントのツールから Cursor の **トークン計測を安定取得できない**ことが多い。`**useCanvasState`** と入力欄で、スクリプト化した各プロンプトのあとにファシリテータが Cursor の利用状況 UI やストップウォッチから **トータルと送信→最終応答までの ms** を **貼り付ける**。**SPL の文字数や推測値を「LLM トークン」と表現しない。**

4. **精度:** 永続フィールドに **PASS/FAIL** と **根拠 1 行**（参照バケットの数値一致、誤 index、欠落フィールド等）を書く。感覚だけの「大丈夫」にしない。

5. **役割分担:** **調査の流れ**は `agent_skills_*.md`、**数値と出所**は Canvas に置き、聴衆が **一画面で根拠**を追えるようにする。
